# Lycanites Structure-Territory Integration — Design

Status: approved (2026-06-16)

## Goal

Integrate Lycanites Mobs into Runic Structures so curated structures feel like
hostile, claimed territory defended by element-themed local ecologies. The work
stays within this mod's server-side purpose: forced-spawn control plus mob
enhancement. Two modules:

1. **Territory pools (data)** — blend element-matched Lycanites mobs into the
   existing per-archetype `mobPool`s, scaled by `progressionTier`; blacklist the
   three Lycanites bosses.
2. **Element-aware guardian theming (code)** — make *elite* Lycanites mobs
   element-themed and tag them as structure guardians. High-signal, low-volume.

Anything outside a server-side spawn mod (spell affinities, client VFX, Treasure
Bags, MCA/Recruits, Runic Skills XP, Lycanites' own `StructureSpawnInjector`
packs) is **out of scope**. We expose the `rs_guardian` tag *for* downstream
systems but do not implement them here.

## Module 1 — Territory pools (data)

### Mechanism

`StructureProfile.mobPool` already drives periodic and initial-population
spawning (`TickSpawnHandler` → `MobSelector.pickFromMobPool`). Adding Lycanites
IDs to an entry's `mobPool` is sufficient; no code change is needed for spawning.
Missing IDs resolve gracefully (`pickFromMobPool` skips unresolved/unsummonable
types), so **no hard dependency on Lycanites is introduced**.

### Generation approach

The defaults file `src/main/resources/defaults/runicstructures-structures.json`
is a ~101 KB single-line minified JSON with hundreds of entries that repeat ~8
archetype pools. It will be regenerated by a **one-shot Python script** (kept in
the repo under `tools/` for reproducibility) that:

1. Parses the existing JSON.
2. Classifies each structure entry by a **signature mob** already present in its
   `mobPool`.
3. Appends the tier-scaled Lycanites IDs for that archetype (deduped, appended
   after existing entries, preserving order).
4. Logs any entry whose pool matches no signature (left unchanged) so coverage is
   auditable.
5. Writes back minified, preserving all other fields and `_section_*`/`_comment`
   keys.

The generator is **idempotent** (it strips existing `lycanitesmobs:` entries from
each pool before re-adding, so it can run on its own output) and **asserts** that
no addition is a Lycanites boss or a `peaceful` creature (peaceful creatures are
CREATURE-category and would spawn passively — not valid hostile guardians). The
`peaceful` flag is read from the Lycanites source at generation time.

### Archetype signatures and additions

Detection runs in this priority order (first match wins) because some pools share
mobs. Two ordering hazards drive the order: ancient city and cave both contain
`creature_compendium:moleman` (ancient city first); and graveyards/crypts contain
BOTH `iceandfire:dread_ghoul` AND `minecraft:wither_skeleton`, so every crypt
signature must precede the blaze-less nether-camp signatures.

An archetype may have several signatures (any one matching classifies the pool).

| Order | Archetype | Signature mob(s) (any one, must be in pool) | Lycanites additions by tier |
|---|---|---|---|
| 1 | Crypt/stronghold | `cataclysm:draugr`, `iceandfire:dread_ghoul`, `irons_spellbooks:catacombs_zombie` | `ghoul, geist, cryptkeeper, necrovore, banshee, reaper, shade, grue` |
| 2 | Nether | `minecraft:blaze` | mid: `afrit, cinder, belphegor, salamander, trite` · late: + `cherufe, behemophet, wraith, malwrath, volcan, khalk` |
| 3 | Ocean/water | `cataclysm:deepling` | early: `skylus, herma, abtu` · mid: + `jengu, ioray, lacedon, roa, thresher` · late: + `abaia` |
| 4 | End | `cataclysm:endermaptera` | (late/veteran) `argus, astaroth, grell, spectre, trite, naxiris` |
| 5 | Ancient city | `minecraft:phantom` | `grue, shade, fear, spectre, banshee, epion, darkling` |
| 6 | Mage/arcane | `ars_nouveau:wilden_guardian`, `irons_spellbooks:archevoker` | `argus, naxiris, grell, sutiramu` (arcane/eldritch guardians) |
| 7 | Nether camps | `minecraft:piglin_brute`, `minecraft:wither_skeleton`, `cataclysm:ignited_berserker` | (same as Nether, default tier `mid`) |
| 8 | Illager | `irons_spellbooks:magehunter_vindicator` | **none** — no thematic match; respects village/illager restraint |
| 9 | Cave/mineshaft | `creature_compendium:moleman` | early: `darkling, gnekk, gorgomite, eyewig` · mid: + `grue, troll, maug, geonach, shade` · late: + `wraamon, tremor` |
| 10 | Early-generic ruins | `variantsandventures:gelid` | `conba, lycosa, ghoul` (light, low-threat) |

Pools matching no signature are left unchanged and logged. With the current pack
this leaves 10 entries untouched — all coherent factions or signature-ambiguous:
the Goblins of Tyranny faction (5), `mowziesmobs:barako_village`,
`ba_bt:land_tower`/`ocean_tower` (mixed cultist), and `mtr:desert_temple`/
`jungle_temple` (no signature distinct from the late `ba_bt` towers). This honors
the doc's faction/village restraint rule.

Notes:
- All IDs are prefixed `lycanitesmobs:` when written.
- Tier scaling is **cumulative**: a `late` entry receives early+mid+late
  additions; a `mid` entry receives early+mid. Entries with no `progressionTier`
  are treated as their archetype's default (nether→mid, ocean→mid, cave→mid,
  crypt/ancient/end→late, early-generic→early).
- The three matched groups for crypt/ancient/end are single-tier sets (those
  structures are inherently late/veteran), so no sub-tier scaling is applied
  there beyond presence.
- Mappings are grounded in Lycanites' actual `creatureType` / `elements` /
  `spawning.spawners` / `biomeTags` data, not the reference doc's prose.

### Boss handling

Add to the **default `mobBlacklist`** in `RSConfig.buildMobConfig` (matching the
existing vanilla/modded boss entries):

```
lycanitesmobs:rahovart, lycanitesmobs:asmodeus, lycanitesmobs:amalgalich
```

This protects them in the force-placement path (`isEntityAllowed`) and in
`pickFromMobPool`. They are never added to any pool.

## Module 2 — Element-aware guardian theming (code)

### Data layer

New bundled classpath resource
`src/main/resources/defaults/runicstructures-lycanites-elements.json`, generated
from the Lycanites source by the same `tools/` script. Shape:

```json
{ "lycanitesmobs:afrit": { "element": "fire", "type": "imp" }, ... }
```

Loaded once into a new `LycanitesData` class (classpath read; not copied to
serverconfig — it is reference data, not user config). Provides:
`getElement(ResourceLocation)`, `getType(ResourceLocation)`, `isLycanites(...)`,
`elementAdjective(String element)`.

For creatures with multiple elements, the **first** listed element is used. For
creatures with no element, theming falls back to the structure's existing
`eliteNamePrefix`.

### Element → adjective table

Covers Lycanites' full elemental palette:

| Element | Adjective | Element | Adjective |
|---|---|---|---|
| fire | Infernal | order | Hallowed |
| lava | Molten | chaos | Chaotic |
| frost | Frozen | light | Radiant |
| water | Tidal | arbour | Thornclad |
| air | Gale | acid | Caustic |
| earth | Stoneborn | aether | Aetherial |
| quake | Tremorous | poison | Venomous |
| lightning | Storm | fae | Feywild |
| shadow | Shadowed | phase | Phantom |
| nether | Hellborn | void | Voidtouched |
| arcane | Arcane | | |

An unknown element string falls back to the structure `eliteNamePrefix`.

### Hook

In the elite application path of **both** spawn routes
(`SpawnEventHandler.onFinalizeSpawn` and `TickSpawnHandler`'s periodic elite
branch), after `MobEnhancer.applyEliteEnhancements`:

- If `RSConfig.lycanitesEliteTheming` is on **and** the mob is a known Lycanites
  creature: set the custom name using the element adjective as the prefix (reuse
  the existing `entity.runicstructures.elite_prefix` translation key →
  "Infernal Afrit"), and add the `rs_guardian` tag (in addition to `rs_elite`).
- This is implemented as a single new method, e.g.
  `MobEnhancer.applyLycanitesGuardianTheming(mob, profile)`, called right after
  the elite enhancement so it overrides the generic name only for Lycanites
  elites. Non-Lycanites elites are unchanged.

### Bonus XP

Extend `RunicStructures.onEliteMobDeath`: when the dead mob has the `rs_guardian`
tag, award an **additional** `RSConfig.guardianBonusXP` on top of the existing
elite XP. Guarded by `eliteSpawnsEnabled` like the existing elite XP.

### Config — new `lycanites` section in `RSConfig`

| Key | Type | Default | Meaning |
|---|---|---|---|
| `eliteTheming` | bool | `true` | Element-themed names + `rs_guardian` tag for Lycanites elites |
| `guardianBonusXP` | int (0–500) | `50` | Extra XP awarded on `rs_guardian` death |

No new gameplay auras or stat tiers — guardians reuse the elite stat/glow/
persistence infrastructure. This keeps the surface minimal and avoids balance
landmines, while giving downstream systems a detectable `rs_guardian` tag.

## Validation / testing

- `./gradlew build` compiles cleanly.
- `LycanitesData` load failure is non-fatal: logs a warning, theming becomes a
  no-op (returns "not Lycanites" for all IDs).
- Extend `RunicStructures.runConfigValidation` to surface a warning if the
  bundled element map failed to load or contains invalid ResourceLocation keys.
- Generator script self-checks: every pool entry matched a signature or was
  logged as skipped; output re-parses as valid JSON; boss IDs are in the
  blacklist and in no pool.
- Manual checklist after build: spawn into a nether/ocean/cave/crypt structure
  and confirm Lycanites types appear in pools; force an elite Lycanites spawn and
  confirm element-themed name + `rs_guardian` tag + bonus XP on kill.
